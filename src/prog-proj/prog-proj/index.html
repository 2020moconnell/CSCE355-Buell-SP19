Self-testing software files

Here is software for you to test your own programs. I will use the same software in the grading process, and so I urge you to use it as well so that you will not have any unpleasant surprises.

There are two .zip files: one unzips to a directory containing various executables; the other unzips to a directory tree with test files, both input and output.

Installing the files

I am assuming that you will use this software either on a GNU/Linux system (such as those available in our Linux labs) or on an Apple Mac running some recent version of OS X (with Perl 5 installed, which is probably is). I can't guarantee that anything will work properly on any other system; indeed, there are some precompiled binaries that presumably will only run on one of the two systems, above.

Download these two .zip files to your system:

  1. either bin-linux.zip or bin-mac-osx.zip, depending on your system, and
  2. test-suite.zip (system-independent)
to your system. The first file unzips to a directory named bin containing a handful of executable files. The second unzips to a directory tree named test-suite containing both input files to feed to your programs and output files to compare with your programs' output. You should be able place these subdirectories virtually anywhere, except DON'T unzip the files where you already have a subdirectory named bin or test-suite if you're at all worried about files being overwritten.

On the command line, move each zip file to the place where you want the corresponding subdirectory to be and type

  unzip <zip-file.zip>

for each zip file.

Then, you will need to hand-edit two lines in the file project-test.pl in the bin directory. On line 19, change the path being assigned to the variable $test_suite_root to be the pathname of your current test-suite directory (depending on where you put it in the previous step). Similarly, on line 22, change the pathname being assigned to the variable $bin_dir to be the pathname of your current bin directory. Note that $ENV{HOME} refers to your home directory. (The current settings of these variables refer to my system.) Save your changes and exit the editor.

That's it. You may find it convenient to add the bin directory to your $PATH variable, but that's optional. The next section is worded assuming you have done this.

Using the files to test your code

You should read this entire subsection before using any of the testing software.

The bin directory contains the file project-test.pl (which you edited). This is the file you will use most often, and it may be the only file you ever use. As with the other .pl files in the directory, it is a Perl script that controls the execution of several other processes, including building and running your programs and testing their output against the solutions. When you run project-test.pl, you pass it the directory containing your source code. The program then moves into this directory and does the following for each of the six possible tasks:

  • looks for the .txt file you provide with instructions on building and running your program for this task, if any (otherwise skips to the next task);
  • attempts to build your program using the build command(s) you provide;
  • runs your program a number of times on various input files found under the test-suite directory, comparing your output with the solutions (also under the test-suite directory).

To invoke the script, type

   project-test.pl <my-source-code-directory>

at the shell prompt.

During its execution, the script puts comments into a file comments.txt in your source code directory. After the script finishes, you should examine this file. It contains routine messages explaining what the script was doing and when, as well as any problems it encountered. Error messages are given in ALL CAPITAL LETTERS so that they are easy to spot. The end of the file contains a summary of what the script found for each task.

You may run the script project-test.pl any number of times. On each run, the comments.txt file is overwritten, but a back-up file comments.bak is created beforehand.

IMPORTANT: please be aware that, besides the comments file, the script modifies your source code directory in other ways. Here is a summary of the script's modifications:

  • It creates any intermediate files used in compilation (such as .o files), depending on your build instructions. (It does not remove these unless your instructions include commands to remove them.)
  • It creates (and does not delete) any executable binaries resulting from compiling your programs.
  • It creates the comments.txt and/or comments.bak files described above.
  • It creates a file errlog.txt containing what you see on the screen as the script executes.
  • It creates a temporary subdirectory test-outputs to hold the outputs of your programs. (This subdirectory is removed upon normal completion of the script.)

Other things you can do

The bin directory contains other files you can use to test and debug your programs. There are three binary executables that are called from the project-test.pl script but that you can also run as stand-alone programs:

  • isDFA reads text from standard input and reports whether or not the input constitutes a correctly formatted DFA. If so, it gives the number of states and the alphabet. For example:

      $ isDFA < some_random_garbage_file.txt
      Not a DFA
      $ ./searcher str8.txt | isDFA
      A DFA with 4096 states and alphabet abcdefghijklmnopqrstuvwxyz

  • DFAiso takes a DFA file as a command line argument (we'll call it the reference DFA) and compares it with another DFA read from standard input (the input DFA). It reports whether or not the two DFAs are isomorphic, that is, whether or not they are the same DFA up to reordering of states. The reference DFA must be sane, that is, all states must be reachable from the start state. If you use the command line switch --loose (with two hyphens), the program will only test isomorphism of the reference DFA with the reachable portion of the input DFA. This only matters if the input DFA is not sane. (There is also a --strict option, but that's the default.) For example,

      $ ./minimizer nonminimalDFA2.txt | DFAiso minDFA2.txt
      The two DFAs are isomorphic.
       0 1
       0 1
      $ DFAiso minDFA3.txt < minDFA5.txt
      Accepting state discrepancy found: transition `0´ from states (0,0)
      The two DFAs are different.
      $ DFAiso minDFA1.txt < nonminimalDFA7.txt
      Sizes differ.  Ref DFA: 5 states; input DFA: 8 states
      The two DFAs are different.
      $ DFAiso --loose minDFA1.txt < nonminimalDFA7.txt
      The two DFAs are isomorphic.
       0 1 2 3 4
       0 2 4 6 7

    If the DFAs are isomorphic, a matching of states between the two DFAs is also displayed.
  • DFAequiv takes two DFA text files as arguments and reports whether or not the DFAs are equivalent, that is, whether or not they recognize the same language. For example,

      $ DFAequiv nonminimalDFA4.txt minDFA4.txt
      The two DFAs are equivalent.
      $ DFAequiv nonminimalDFA4.txt minDFA5.txt
      The two DFAs are not equivalent.

The other files in the bin directory are Perl scripts used to generate and process test input files. Comments near the top of these files explain what they do and how to use them. All their output is to standard output.

You can copy and edit any of these files, including project-test.pl. For example, you can suppress the deletion of the test-outputs subdirectory and its files by setting the $delete_temps flag to 0 on line 75, and you can adjust the execution time-out limit on line 64.
